# DATABASE_RULES.md

# HMS v2 Database Engineering Standards

---

# Purpose

Defines all database standards for HMS v2.

Every schema, migration and query must follow these rules.

---

# Database Philosophy

The database is the single source of truth.

Never duplicate information unnecessarily.

Normalize appropriately while keeping practical performance in mind.

---

# Primary Database

PostgreSQL

ORM

Prisma

---

# Primary Keys

Use UUIDs internally for primary keys.

Human-readable IDs (UHID, OPD ID, IPD ID, Invoice Number, etc.) should be stored as separate unique fields generated by business logic.

Never expose internal UUIDs to users unless required for technical APIs.

# Money Rules

Never use floating-point numbers for monetary values.

Store currency values using fixed-precision decimal types (for example, `DECIMAL(12,2)` or the Prisma `Decimal` type) to avoid rounding errors.

---

# Date Rules

Always store UTC.

Convert to local timezone in UI.

---

# Concurrency Rules

Prevent

Duplicate OPD

Duplicate Billing

Duplicate Invoice

Duplicate Refund

Duplicate UHID

Handle concurrent requests using database transactions and appropriate constraints.

---

# Query Rules

Never use SELECT *

Always select required fields.

Use pagination.

Optimize joins.

Avoid N+1 queries.

---

# Prisma Rules

No raw SQL unless absolutely necessary.

Prefer Prisma Client.

Document every migration.

Never modify production migrations.

---

# Performance Rules

Index foreign keys.

Index search fields.

Batch inserts where appropriate.

Avoid unnecessary queries.

---

# Backup Rules

Daily backup.

Restore testing.

Migration versioning.

---

# Future Compatibility

Database must support

Electron

Multiple deployments

Future integrations

API extensions

Without requiring structural redesign.
